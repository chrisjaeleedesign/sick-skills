# Double Diamond Flow

A structured exploration that moves through problem understanding to solution design.

## When to use

The user has a complex problem that needs both problem exploration and solution exploration before acting. Good for feature design, system architecture, or any situation where jumping to solutions too early would be risky.

## Steps

### 1. Discover (go broad on the problem)
Explore the problem space. Ask open questions: What's the context? Who's affected? What's been tried? What are the constraints? Use ask calls to get outside perspectives on the problem framing.

**Transition:** When the problem space feels well-explored, announce: "I think we have a good picture of the problem space. Ready to narrow down?"

### 2. Define (narrow to the core problem)
Converge on a clear problem statement. What specifically are we solving? What's in scope vs out? What does success look like?

**Transition:** When there's a crisp problem statement, announce: "Here's the problem we're solving: [statement]. Ready to explore solutions?"

### 3. Develop (go broad on solutions)
Explore solutions. Generate multiple approaches — use a wide flow here if helpful. Don't judge yet, just collect options. Call ask with different models/personas for diverse ideas.

**Transition:** When there are 2-4 solid options on the table, announce: "We have a few good options. Ready to pick one?"

### 4. Deliver (narrow to a recommendation)
Converge on an approach. Compare the options against the success criteria from Define. Make a recommendation with clear reasoning.

## Tips

- Each phase can be as short or long as the problem warrants. Simple problems might spend one exchange per phase. Complex ones might need several.
- The user can skip phases or go back. "Let's go back to Discover" is always valid.
- The value is in the diverge-then-converge rhythm, not in rigidly following four boxes.
