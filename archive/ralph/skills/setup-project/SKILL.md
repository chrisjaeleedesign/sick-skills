Set up this Wiggum Loop project from the user's idea.

User's request: $ARGUMENTS

Follow this workflow exactly:

## Step 1: Detect Context

Determine what we're working with:

**Check for existing source code:**
- Look for non-template source files (anything beyond the default template files like `SPECS/project_spec.md`, `IMPLEMENTATION_PLAN.md`, `AGENTS.md`, `ralph_prompts/`, `loop.sh`, `.claude/`, `.agent/`)
- If real source files exist (e.g., `src/`, `app/`, `lib/`, `*.py`, `*.ts`, `package.json` with real deps) → **augmenting** an existing project. Read the code to understand the stack and conventions.
- If only template files exist → **new project**

**Determine mode from the user's description:**
- If the idea describes a **fixed deliverable** with a clear "done" state (e.g., "build a portfolio site", "create a CLI tool", "make a todo app") → **build mode**
- If the idea describes **open-ended exploration** with no fixed end state (e.g., "experiment with trading strategies", "optimize sorting algorithms", "explore ML model architectures", "iterative research on...") → **experiment mode**
- If unclear, ask the user: "Is this a build project (clear deliverable, done when shipped) or an experiment (open-ended, iterating indefinitely)?"

## Step 2: Gather Requirements

Ask the **minimum** questions needed. Be opinionated — suggest defaults and let the user override.

**For build mode:**
1. Confirm what they're building (use $ARGUMENTS if sufficient)
2. Suggest a tech stack based on what they described. Only ask if ambiguous.
3. Clarify scope — suggest Phase 0 (setup) + 2-3 feature phases. Ask "Does this scope look right?"

**For experiment mode:**
1. Confirm the domain/hypothesis (use $ARGUMENTS if sufficient)
2. Ask: What are the key metrics or evaluation criteria? (suggest defaults based on domain)
3. Ask: Any constraints or beliefs to encode? (e.g., "only test during market hours", "must beat O(n log n)")
4. Suggest a stack suited to the experiment. Only ask if ambiguous.

**For augmenting an existing project:**
- Read existing source files to derive the tech stack and conventions
- Skip stack questions — derive from code
- Focus questions on what new feature/experiment to add

**Skip questions entirely if $ARGUMENTS provides enough context.** Bias toward action.

## Step 3: Generate Files

Generate ALL of the following files. Write them completely — no placeholders or TODOs.

### 3a. SPECS

**Build mode** → Write `SPECS/project_spec.md`:
```
# [Project Name] — Specification

## Overview
[What this project does, in 2-3 sentences]

## Core Features
- [Feature 1]: [Brief description]
- [Feature 2]: [Brief description]
...

## Tech Stack
- [Framework/language]
- [Key dependencies]

## Non-Goals
- [What this project intentionally does NOT do]
```

**Experiment mode** → Write `SPECS/experiment_spec.md`:
```
# [Experiment Name] — Specification

## Hypothesis
[What we're exploring and why]

## Evaluation Criteria
- [Metric 1]: [How measured, what counts as success]
- [Metric 2]: [How measured, what counts as success]

## Constraints
- [Hard constraint 1]
- [Hard constraint 2]

## Tech Stack
- [Framework/language]
- [Key dependencies]

## Domain Context
[Background knowledge the loop agent needs to do good work]
```

### 3b. IMPLEMENTATION_PLAN.md

Overwrite the existing template with a real phased plan:

```
# Implementation Plan

> **Ralph Loop Rule**: Do exactly ONE unchecked task per iteration. If this plan drifts, regenerate from specs — do not patch endlessly.

---

## Phase 0 — Setup

- [ ] [Specific setup step, e.g., "Initialize Next.js project with TypeScript"]
    - [Details: commands to run, files to create]
- [ ] [Another setup step if needed]
- [ ] **Verify**: [How to confirm setup works, e.g., "Dev server starts on localhost:3000"]

## Phase 1 — [First Feature/Experiment Phase]

> **Goal**: [What done looks like for this phase]

- [ ] **[Area]**: [Actionable task]
    - [Technical details]
- [ ] **[Area]**: [Another task]
    - [Technical details]
- [ ] **Verify**: [Specific verification]

[Add Phase 2, Phase 3, etc. as needed — 3-7 tasks per phase]

---

## Changelog

_Updated by Ralph loop iterations. Each completion adds an entry below._
```

**Build mode**: Plan all phases up front (typically 3-5 phases total).
**Experiment mode**: Plan Phase 0 (setup) + Phase 1 (first experiment batch, 2-4 tasks). Later phases are generated by EXPERIMENT_NEXT.md.

### 3c. AGENTS.md

Overwrite with conventions matching the chosen stack:

```
# Agent Conventions

## Commands

- **Dev Server**: [actual command]
- **Test**: [actual command]
- **Lint**: [actual command]
- **Build**: [actual command]

## Code Style

- [Language/framework conventions]
- [Component patterns]
- [Naming conventions]

## File Structure

- `/src`: [Description]
- `/tests`: [Description]
- `/SPECS`: Project specifications
- [Other relevant directories]
```

### 3d. EXPERIMENT_NEXT.md (experiment mode only)

Overwrite `ralph_prompts/EXPERIMENT_NEXT.md` with a domain-specific version tailored to this experiment. The prompt should:
- Read SPECS and COMPLETED.md to understand what was learned
- Read the current IMPLEMENTATION_PLAN.md
- Analyze results from the completed phase
- Generate the next experiment phase (2-4 tasks) appended to the plan
- Focus tasks on the most promising direction based on results

## Step 4: Handover

Summarize what was configured:

**Build mode:**
> Project configured! Here's what I set up:
> - **Spec**: SPECS/project_spec.md — [brief summary]
> - **Plan**: IMPLEMENTATION_PLAN.md — [N phases, M total tasks]
> - **Conventions**: AGENTS.md — [stack summary]
>
> Run `./loop.sh` to start building.

**Experiment mode:**
> Experiment configured! Here's what I set up:
> - **Spec**: SPECS/experiment_spec.md — [brief summary]
> - **Plan**: IMPLEMENTATION_PLAN.md — Phase 0 (setup) + Phase 1 (first experiment batch)
> - **Conventions**: AGENTS.md — [stack summary]
> - **Experiment driver**: ralph_prompts/EXPERIMENT_NEXT.md — generates new phases after each batch completes
>
> Run `./loop.sh --experiment` to start exploring. The loop will keep generating new experiment phases as it learns.
